### REST vs GraphQL vs gRPC
- **REST (Representational State Transfer):** REST APIs or Representational State Transfer Application Programming Interfaces, are a set of rules for building web 
services that allow different applications to communicate over the internet. They use standard HTTP methods like GET, POST, PUT, and DELETE to perform operations on 
resources, making them flexible and easy to use
- **GraphQL:** GraphQL is a data query and manipulation language developed by Facebook that allows clients to request exactly the data they need from APIs.
It solves the over-fetching problem of REST and is great for complex front-end applications. It uses a strongly-typed schema to define the structure of the data, 
making APIs more flexible and predictable.
- **gRPC:** GRPC is an open-source framework developed by Google for remote procedure calls (RPC), allowing different services to communicate efficiently over a 
network. It uses HTTP/2 for transport and Protocol Buffers for data serialization, making it suitable for modern applications, especially in microservices architectures.

### API Authentication Methods
- **Basic Auth:** The simplest method, passing a Base64 encoded username and password in the HTTP header. It is highly insecure unless used strictly over HTTPS.
- **API Keys:** A unique string generated by the server and passed by the client (usually in a header or query parameter) to identify the calling project or user.
Easy to implement but difficult to secure if the key is exposed.
- **Bearer Tokens (JWT):** JSON Web Tokens are stateless tokens that contain encrypted "claims" (user data, roles, expiration) within the token itself. Widely used 
for modern web and mobile app sessions.
- **OAuth 2.0:** A standardized authorization framework (not an authentication protocol natively) that allows third-party applications to obtain limited access to 
an HTTP service without sharing credentials. 
- **mTLS (Mutual TLS):** Both the client and the server verify each other's digital certificates before establishing a connection. Common in high-security environments like 
banking or Zero Trust architectures.

### API Rate-Limiting Algorithms
- **Token Bucket:** Tokens are added to a "bucket" at a constant rate. Each API request consumes a token. If the bucket is empty, requests are dropped. This allows 
for sudden bursts of traffic as long as tokens are available.
- **Leaky Bucket:** Requests enter a queue (the bucket) and are processed at a strictly constant rate. If the bucket overflows, new requests are discarded. This smooths 
out traffic into a steady stream.
- **Fixed Window:** A counter tracks requests within a fixed time window (e.g., 00:00 to 00:01). It resets at the end of the window. This can cause traffic spikes right 
at the boundary of a reset.
- **Sliding Window Log:** Stores a timestamp for every request and dynamically calculates the limit based on the exact previous timeframe. Highly accurate but consumes 
significant memory.

### API Versioning Strategies
- **URL Pathing:** Including the version directly in the URL (e.g. `api.example.com/v1/users`). It is the most common and easiest to route, though some argue it 
violates strict REST principles.
- **Query Parameters:** Passing the version as a variable (e.g.`/users?version=1`). Easy to implement, often used for minor filtering rather than major structural changes.
- **Custom Headers:** Using a specific header to dictate the version (e.g.`X-API-Version: 2`). Keeps URLs clean, but makes it harder to test endpoints directly in a web browser.
- **Content Negotiation (Accept Header):** Modifying the standard Accept header (e.g.`Accept: application/vnd.company.v1+json`). This is the most strictly RESTful 
approach, but can be complex for clients to implement correctly.

### API Security Best Practices
- **Enforce HTTPS:** Always use TLS/SSL to encrypt data in transit and prevent Man-in-the-Middle (MitM) attacks.
- **Input Validation & Sanitization:** Never trust client data. Validate payloads strictly to prevent SQL injection and Cross-Site Scripting (XSS).
- **Implement Rate Limiting & Throttling:** Protect your server from DDoS attacks and brute-force attempts by capping request volume.
- **Principle of Least Privilege:** Ensure tokens or keys only grant the minimum permissions (scopes) necessary for the task, utilizing RBAC (Role-Based Access Control).
- **Use an API Gateway / WAF:** Place your APIs behind a Web Application Firewall or Gateway to handle authentication, rate limiting, and malicious traffic filtering centrally.
- **Avoid Sensitive Data in URIs:** Never put passwords, API keys, or sensitive PII in the URL path or query parameters, as these get logged by web servers and proxies
use the POST body instead.
