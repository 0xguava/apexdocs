### Multithreading vs Multiprocessing

- **Multithreading:** Runs multiple threads concurrently within a single process. All threads share the same memory space. In standard Python (CPython), the Global 
Interpreter Lock (GIL) prevents threads from executing Python bytecodes simultaneously. It is best used for **I/O-bound tasks** (e.g., downloading files, making web requests).
- **Multiprocessing:** Creates entirely new, independent processes, each with its own memory space and Python interpreter. This bypasses the GIL and allows true simultaneous 
execution. It is best used for **CPU-bound tasks** (e.g., heavy mathematical computations, image processing).

### Thread Pool: Who uses it?
A Thread Pool (such as Python's `concurrent.futures.ThreadPoolExecutor`) is a design pattern that maintains a collection of pre-instantiated, idle threads ready to be given tasks.
This avoids the performance overhead of constantly creating and destroying new threads. It is primarily used by backend developers, data engineers, and network programmers.
They use thread pools to efficiently manage a large volume of concurrent I/O-bound tasks—like making hundreds of external API calls, executing bulk database queries,
or mass web scraping—without overwhelming the system's resources.

### Concurrency vs Parallelism

- **Concurrency:** The ability of a system to handle multiple tasks at once by rapidly switching between them. It is about *dealing* with multiple things at once 
(e.g., a single chef taking turns chopping veggies, boiling pasta, and frying chicken).
- **Parallelism:** The ability of a system to execute multiple tasks at the exact same physical time, usually across multiple CPU cores. It is about *doing* multiple 
things at once (e.g., three different chefs, each handling one dish simultaneously).

### Unit Testing: Define Test Cases
A test case is the smallest, individual unit of testing in software development. In Python (using frameworks like `unittest` or `pytest`), a test case is a specific function 
written by a developer to verify that a tiny, isolated piece of code (like a single function or method) behaves exactly as expected given a specific set of inputs. 
It uses "assertions" to check if the actual output matches the expected outcome.

### Docstrings vs Comments
- **Comments:** Written using the `#` symbol. They are meant for other developers reading the source code to explain *why* or *how* a specific block of code works
. Python ignores comments entirely during execution.
- **Docstrings:** Written using triple quotes (`"""`). They are meant to document *what* a module, class, or function is supposed to do. Unlike comments, docstrings are 
attached to the object at runtime (accessible via `__doc__`) and are used by IDEs and automated documentation generators (like Sphinx).

### Ruff and Black: Git Hooks to Test Python Code
- **Black:** An uncompromising, opinionated code formatter that automatically rewrites your Python code to ensure a uniform style across your entire project.
- **Ruff:** An extremely fast Python linter (written in Rust) that scans your code to catch syntax errors, unused variables, and bad practices before they cause bugs.
- **Git Hooks:** Developers use tools like `pre-commit` to automatically run Black and Ruff as "hooks" right before code is committed to Git. If Ruff finds an error 
or Black needs to reformat the code, the commit is blocked until the issues are fixed, ensuring only clean code reaches the repository.

### Standard Python File Format
To maintain readability and adhere to PEP 8 standards, a typical Python script is organized in the following top-to-bottom order:
- **Imports:** Standard library modules (e.g., `import sys`)
- **Import Functions:** Third-party and local module functions (e.g., `from requests import get`)
- **Declare Constants:** Global constant variables, usually in all caps (e.g., `MAX_RETRIES = 5`)
- **Classes:** Object-oriented class definitions.
- **And Rest:** Helper functions, followed by the main execution block (`if __name__ == "__main__":`).
